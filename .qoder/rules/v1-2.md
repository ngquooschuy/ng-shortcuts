# Core Behavior Rules

## 1. Space-separated tokens mean sequence

```text id="b7g3t0"
g h
g d
up up down down
```

## 2. Combo tokens still supported

```text id="2hjjy1"
shift+g
mod+k
alt+1
```

## 3. Timeout reset

If user pauses too long between sequence steps, reset sequence.

Default:

```ts id="2d4j7r"
1000ms
```

Per binding override:

```ts id="j0s0ei"
sequenceTimeout: 1500
```

## 4. Exact order only

`g h` matches only:

```text id="m2t58p"
g then h
```

Not reverse.

## 5. Partial progress allowed

If user presses:

```text id="0fl8im"
g
```

Then engine waits for next valid token.

## 6. Wrong key resets sequence

# Design Goal: Minimal Sequence Engine

Implement a tiny internal engine.

Suggested idea:

## Parse hotkey into steps

```text id="7ijg6m"
'g h'
=> ['g', 'h']

'up up down down'
=> ['up','up','down','down']
```

Each step may itself be combo:

```text id="6fg58z"
'shift+g h'
=> ['shift+g', 'h']
```

## Runtime State

Maintain only:

```ts id="3shw58"
currentIndex
lastTimestamp
```

No giant architecture.

---

# Sequence Matching Rules

Given registered:

```ts id="7p4e5x"
g h
g d
```

If user presses:

```text id="g08hhq"
g
```

Engine should wait.

Then:

```text id="s7v3oq"
h
```

=> trigger `g h`

Then reset.

If next key invalid:

```text id="z3jv91"
x
```

=> reset.
